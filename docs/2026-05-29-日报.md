# 日报 2026-05-29

## 一、Phase 1 低价值邮件展示改造

**问题**：Phase 1 判定为不重要的邮件被静默丢弃，用户看不到。随着高频使用，未读邮件越积越多。

**方案**：Phase 1 不再丢弃低价值邮件，改为生成一张折叠的 cleanup bundle 卡片。

**实施**：
- `phase1.py`：`is_candidate` 拆为 `user_action: reply | review | ignore`；ignore 走 cleanup bundle，reply/review 走 Phase 2
- `storage_types.py`：`PersistentCard` 新增 `card_type`、`bundled_messages`、`user_action` 字段
- `card_service.py`：新增 `build_cleanup_bundle()`，零 LLM 成本，header 信息直出；`merge_cards` 丢弃旧 cleanup 卡片
- `pipeline.py`：Phase 1 结果解构为 candidates + low_value_items 两路
- `app.js`：cleanup bundle 渲染（折叠/展开、批量已读逐行动效、计数器）
- `style.css`：cleanup 卡片完整样式

## 二、item_type 映射链路重构

**问题**：Phase 1 LLM → 15 种 item_type → CandidateKind → Phase 2 LLM → base_judgment.item_type → 前端 ITEM_TYPE_CATEGORY，四层映射层层漂移。qwen3-max 存在 association bias：判了 `AccountNotice` 就内在绑定低优先级，导致 reason 写"useful signal"但 user_action 输出 "ignore"。

**方案**：让 LLM 直接输出 `user_action`，`item_type` 退化为信息标签，不再驱动分类逻辑。最终彻底删除 `item_type`，分类全由 `user_action` 驱动。

**实施**：
- `phase1.py`：
  - `_PHASE1_SYSTEM` 删除 `## Item Types` 段
  - 删除 `_ITEM_TYPE_TO_KIND` 15 种映射表，改为 `_user_action_to_kind()` 三行推导
  - `parse_phase1_response` 不读 `item_type`，candidate_kind 从 user_action 推导
  - reason 指令改为 "Explain WHY you chose this user_action"
  - 加 Self-check 规则："输出前重读 reason，如果说了 useful/awareness/signal/deadline → 不能是 ignore"
  - ignore 定义收紧：删除 "interview reminder/new-application" 等误导性示例，review 明确列出正向示例，加 "if in doubt → review"
- `judgment.py`：
  - 删除 `_derive_item_type()` 60 行函数，改为 `_user_action_to_item_type()` 4 行
  - `build_judgment_prompt` / `build_anna_single_judgment_prompt` 输出删除 `item_type`
  - `parse_judgment_output` / `_parse_compact_batch_item`：item_type 从 user_action 代码推导
  - `_enforce_consistency`：规则改为 user_action 驱动
- `card_service.py`：`build_card` / `build_cleanup_bundle` 适配
- `app.js`：`cardCategory` 改用 `userAction` 单字段；cleanup/review tab 不过滤 lower 卡
- `types.py`：`FinalDecision` 新增 `user_action` 字段

## 三、两条 LLM provider 路径差异分析

**发现**：DashScope 和 Anna Sampling 在 Phase 1/2 存在 10+ 处差异，包括：
- Phase 1：批大小（全量 vs 8封）、fallback 策略（允许 vs 不允许降级）
- Phase 2：prompt 详细度（few-shot+rubric vs 极简扁平）、max_tokens（1024 vs 1200）、timeout（240s vs 60s）、max_attempts（3 vs 1）
- 消息格式、JSON 约束方式、system_prompt 语言（中文 vs 英文）、temperature 默认值、模型选择等

所有差异已记录在对话中（待落文档）。
