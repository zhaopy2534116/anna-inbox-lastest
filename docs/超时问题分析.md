# Anna App 超时问题分析

## 一、协议规范中的超时定义

来自 [Executa Protocol Spec](https://github.com/whtcjdtc2007/anna-executa-examples/blob/main/docs/protocol-spec.zh-CN.md)：

| 方法 | 默认超时 | 说明 |
|------|---------|------|
| `describe` | 5 秒 | 启动后首次调用 |
| `health` | 3 秒 | 周期性检查 |
| `invoke` | **60 秒** | 工具执行（可配置） |

> 规范中的重要注释：invoke 超时是 **"可配置"** 的。但规范中**没有说明配置位置**——`executa.json` 中并不包含 `timeout` 字段，Anna harness CLI 参数中也未见相关选项。

### 2.2 插件内部超时

`main.py` 中 `handle_invoke` 对不同工具设置了不同的 `future.result(timeout=...)`：

| 工具 | 插件内部超时 | 说明 |
|------|------------|------|
| `start_mail_agent_run` | 无（同步返回） | 立即返回 run_id，后台 pipeline 在 event loop 上异步执行 |
| `get_mail_agent_run` | 无（同步返回） | 从内存字典读取状态 |
| V2 交互工具 | **120 秒** | `get_active_cards`, `get_card_detail` 等 |
| `run_mail_agent` | **600 秒** | 直接同步等待 pipeline 完成（非 async 模式） |

但**插件内部超时没有意义**——因为 Anna harness 的 invoke 超时（60 秒）会先触发，插件的 120 秒 timeout 永远不会被用到。

## 三、执行流程追踪

### 3.1 前端调用链（Scan now 按钮）

```
用户点击 "Scan now"
  → startScan()
    → invokeTool("start_mail_agent_run", { ai_provider: "dashscope", ... })  ← ① 关键
    → for (poll < 30) { invokeTool("get_mail_agent_run", ...) }              ← ② 轮询
    → loadActiveCards() → invokeTool("get_active_cards", ...)                ← ③ 刷新
```

### 3.2 后端执行链（start_mail_agent_run）

```
main thread (stdin reader)
  → handle_line → handle_request → handle_invoke
    → start_mail_agent_run()
      → MAIL_AGENT_RUNS[run_id] = {...}
      → asyncio.run_coroutine_threadsafe(run_mail_agent_background(...), loop)
      → return { run_id, status: "queued" }  ← 毫秒级返回 ✅

event loop thread
  → run_mail_agent_background()
    → run_mail_agent_pipeline()
      → run_mail_task(input_, sampling_create_message=None)  ← ai_provider="dashscope"
        → Phase 1: run_phase1_batch_classify()  ← DashScope LLM: ~5秒
        → Context reads: read_candidate_context() × N  ← Gmail缓存读取: ~2秒
        → Phase 2: evaluate_item() × N, 2并发     ← DashScope LLM: ~25-50秒
        → _persist_run_results()                  ← 存储持久化: ~5-10秒
        → generate_action_plan()                  ← 聚合: ~1秒
        总计: ~40-70秒
```

### 3.3 线程模型

```
┌─────────────────────┐     ┌──────────────────────┐     ┌────────────────────┐
│   Main Thread       │     │  ThreadPool (4 workers)│     │  Event Loop Thread │
│   (stdin reader)    │     │  (handle_line)        │     │  (daemon)          │
├─────────────────────┤     ├──────────────────────┤     ├────────────────────┤
│ for line in stdin:  │────→│ handle_request()      │     │ loop.run_forever() │
│   pool.submit(line) │     │   handle_invoke()     │     │   run_mail_agent   │
│                     │     │     start_mail_agent  │     │   _background()    │
│                     │     │       → dispatch to ──│────→│     pipeline()     │
│                     │     │         event loop    │     │     DashScope LLM  │
│                     │     │       → return result │     │     (via asyncio.  │
│                     │     │                       │     │      to_thread)    │
│ sampling response ←─│─────│ sampling.dispatch() ──│────→│     persist()      │
│   (stdin read)      │     │   → resolve future    │     │                    │
└─────────────────────┘     └──────────────────────┘     └────────────────────┘
```

**关键发现：DashScope HTTP 调用通过 `asyncio.to_thread()` 在线程池中执行（`llm.py:217-224`），不会阻塞 event loop。**

## 四、根因分析

### 4.1 直接原因：`ai_provider` 硬编码为 `"dashscope"`

**`app.js:542`**：
```javascript
ai_provider: "dashscope",  // ← 硬编码！
```

这导致即使用户在 Anna harness 环境下运行，pipeline 也走 DashScope 路径而非 Anna Sampling 路径：

```python
# run_mail_agent_pipeline (main.py:778-790)
sampling_create_message = None
if provider == "anna-llm":  # ← 永远不会为 True，因为前端传的是 "dashscope"
    sampling_create_message = sampling_create_message_with_invoke
```

Anna Sampling 路径的优势：
- 由 Host 处理 LLM 调用，不走插件 HTTP
- Host 可能对 sampling 有独立的超时策略
- 单次 batch call 代替多次单条 DashScope 调用

### 4.2 时间瓶颈：DashScope 单条评估

Pipeline 中 Phase 2 使用 `evaluate_item()` 逐条评估，2 并发（`EVALUATE_CONCURRENCY=2`）。10 个候选 = 5 批 × 5-10 秒/批 = 25-50 秒。每个 `evaluate_item` 调用经过 `asyncio.to_thread` → `_call_dashscope_json` → `urllib.request.urlopen(timeout=60.0)`。

单个 DashScope 调用超时为 60 秒，但如果阿里云 API 响应慢（冷启动、限流），20-30 秒很正常。

### 4.3 为什么 `start_mail_agent_run` 是异步的却仍然超时

`start_mail_agent_run` 本身在毫秒级返回（它只创建 run_id 并 dispatch 后台任务）。**Anna harness 的 invoke 超时不应该在这里触发。**

可能的解释：
1. **Anna harness 对 Executa 进程有整体健康检查超时**——如果进程在 60 秒内没有响应任何 RPC（因为 event loop 忙于 DashScope 调用），harness 可能认为进程死锁
2. **`get_active_cards` 在 pipeline 运行期间被调用**——当页面加载时 `refreshCards(true)` 调用 `get_active_cards`，此时如果 pipeline 正在 event loop 上运行，`get_active_cards` 需要等待 event loop。但 DashScope 调用通过 `asyncio.to_thread` 执行，event loop 不应被阻塞
3. **前端的 `start_mail_agent_run` invoke 调用本身通过了 Anna harness，但 harness 在等待响应的过程中经历了某种超时**

### 4.4 最可能的根因

**Harness 的 `tools.invoke` RPC 超时机制不仅用于单个 invoke，还可能适用于整个 Executa 进程的生命周期。** 或者 **`start_mail_agent_run` 的 invoke 在 harness 侧被错误地关联到了后续的后台任务执行时间**

从用户提供的时间戳 `17:25:28.918` 来看，需要对照 `start_mail_agent_run` 的发起时间和 poll 循环的结束时间来判断是否是轮询中的某个 `get_mail_agent_run` 或 `get_active_cards` 调用超时。

## 五、解决方案（按优先级排序）

### 方案 A：将 `ai_provider` 改为 `"anna-llm"`（在 Anna harness 下使用 Anna Sampling）

**改动位置**：`app.js:542`
```javascript
// 改为动态判断
ai_provider: state.runtime.connected ? "anna-llm" : "dashscope",
```

**效果**：
- Anna Sampling 路径下，LLM 调用由 Host 处理，不走插件 HTTP
- Host 对 sampling 有独立的超时机制，可能更宽容
- Pipeline 使用 batch 路径，只需 1 次 Anna Sampling 调用代替多次 DashScope 调用

**风险**：需要在 `manifest.json` 的 `host_capabilities` 中声明 `llm.sample`，并确认 Host 支持 v2 协议协商（当前 `executa.json` 中已声明 `"host_capabilities": ["llm.sample"]`）

### 方案 B：配置 invoke 超时

如果在 `executa.json` 或 harness CLI 中找到超时配置项，将 invoke 超时从 60 秒增加到 180 秒或更长。

需要确认配置方式——可能需要在 Harness 侧查找文档或源码。

### 方案 C：优化 pipeline 速度

1. 将 `EVALUATE_CONCURRENCY` 从 2 提高到 4（减少 DashScope 调用等待时间）
2. 将 `_persist_run_results` 延迟到 invoke 返回后再执行
3. Phase 1 使用纯规则（跳过 LLM 调用），仅在 Anna harness 路径下

### 方案 D：确认 Async 模式是否正常工作

验证 `start_mail_agent_run` → `get_mail_agent_run` 的 async 模式在实际 harness 环境下是否按预期工作。如果 harness 对 `start_mail_agent_run` 的 invoke 有额外的等待逻辑，需要改用其他方式。

## 六、建议的验证步骤

1. **确认超时发生在哪个 invoke 调用** —— 查看完整的 RPC 日志，对比 `start_mail_agent_run` 发起时间和超时时间
2. **先尝试方案 A** —— 改动最小，最符合 Executa v2 的设计理念
3. **如果方案 A 不够，再尝试方案 B** —— 查找 invoke 超时配置位置
