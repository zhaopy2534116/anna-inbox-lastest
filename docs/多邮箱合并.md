# 优化：Thread 内多封未回复收信的关注点合并

## 问题

`_dedupe_by_thread` (pipeline.py:113) 对每个 thread 只保留**最新一封 INBOX**，丢弃其余。当对方连发多封诉求不同的邮件、用户都没回复时，旧邮件的关注点会丢失。

```
A 连发 4 封，用户都没回：
  INBOX_1: "帮我审一下这份文档"
  INBOX_2: "Q1 周报发我一下"
  INBOX_3: "下周的会议你定个时间"
  INBOX_4: "客户在催 deadline 了，急"

当前 → 卡片只反映 INBOX_4，前 3 封的诉求全部丢失
期望 → 一张卡片覆盖 4 封邮件的所有关注点
```

## 设计原则

- 在代码层合并，不推给 LLM 判断
- 保持"一个 thread 一张卡片"的模型
- 合并后的数据沿现有 pipeline 自然流动，不新增特殊分支

## 方案

改动集中在 `pipeline.py` 的两个位置：

### 改动 1: `_dedupe_by_thread` — 保留全部 INBOX

```python
# 改前 (line 137-138):
if inbox_msgs:
    result.append(_max_by_date(inbox_msgs))

# 改后:
if inbox_msgs:
    result.extend(inbox_msgs)
```

每个 thread 的全部 INBOX 消息都进入下一阶段。

### 改动 2: 新增 `_merge_thread_entries` — 合并同 thread 消息

在 `_dedupe_by_thread` 之后、Phase 1 之前调用。

```python
def _merge_thread_entries(messages: list[Any]) -> list[Any]:
    """同一 thread 的多条消息合并为一条。

    合并后的条目保留最新消息的元数据（from/date/message_id），
    snippet 拼接全部消息的 subject + snippet，用序号标记。
    Phase 2 会从缓存读取完整 thread 上下文，所以这里只需拼接摘要。
    """
    from collections import defaultdict

    thread_groups: dict[str, list[Any]] = defaultdict(list)
    for m in messages:
        tid = m.thread_id or m.message_id
        thread_groups[tid].append(m)

    result = []
    for msgs in thread_groups.values():
        msgs_sorted = sorted(msgs, key=lambda m: int(getattr(m, 'internal_date', '0') or 0))
        if len(msgs_sorted) == 1:
            result.append(msgs_sorted[0])
        else:
            latest = msgs_sorted[-1]
            parts = []
            for i, m in enumerate(msgs_sorted, start=1):
                s = (m.subject or "")[:80]
                sn = (m.snippet or "")[:80]
                parts.append(f"[{i}/{len(msgs_sorted)}] {s}: {sn}")
            combined_snippet = " | ".join(parts)
            object.__setattr__(latest, 'snippet', combined_snippet)
            result.append(latest)

    return result
```

### 调用位置

```python
# pipeline.py:248-249 附近
new_messages = _dedupe_by_thread(new_messages)
new_messages = _merge_thread_entries(new_messages)  # ← 新增
```

### 不改什么

| 模块 | 原因 |
|------|------|
| phase1.py | 不变。合并后的条目逐个进入 Phase 1，仍是 1 条/thread |
| context.py | 不变。`get_thread_context` 从缓存读取全部消息 |
| judgment.py | 不变。Phase 2 已能看到完整 thread 上下文 |
| card_service.py | 不变。`merge_cards` 按 thread_id 覆盖，1 张 thread 1 张卡 |

## 数据流

```
扫描 → filter_unprocessed → _dedupe_by_thread (保留全部 INBOX)
     → _merge_thread_entries (同 thread 合并为一条，snippet 拼接)
     → Phase 1 (分类合并后的条目)
     → read_candidate_context (从缓存读完整 thread 上下文)
     → Phase 2 (评估，自然看到全部消息)
     → 一张卡片 (覆盖全部关注点)
```

## 为什么在 Phase 1 之前合并，而不是在 Phase 2

1. **代码层确定性**：合并逻辑是纯字符串拼接，不依赖 LLM 判断
2. **不改变下游**：Phase 1 本来就是一个条目一个 thread，合并后只是条目内容更丰富
3. **Phase 2 天然受益**：`get_thread_context` 从缓存读出全部消息，LLM 看到的 thread 上下文是完整的

## LLM 的角色

LLM 不是在"判断哪些是不同关注点"——代码层已经通过 `_merge_thread_entries` 把同 thread 的 N 封 INBOX 合并成了一条。各阶段 LLM 做的事情没有变：

1. **代码层合并**：`_merge_thread_entries` 把 N 封 INBOX 的 subject+snippet 拼成一条摘要，如 `[1/4] 审文档: 请审一下附件... | [2/4] 周报: Q1周报发我... | ...`
2. **Phase 1**：看到合并后的摘要，判断这个 thread "需要回复"，升级为 `thread_context` 深度
3. **Phase 2**：`get_thread_context` 从缓存读出全部 4 封邮件的完整正文，LLM 自然看到所有诉求
4. **卡片生成**：LLM 面对完整的 thread 上下文，不需要任何特殊指令，自然生成覆盖多个关注点的卡片

所以 LLM 做的是它本来就会做的事——根据上下文生成卡片。合并不是推给 LLM 判断的，是在 `_merge_thread_entries` 里用字符串拼接完成的。

## 验证方式

1. 确认同一 thread 有多封 INBOX 时 `_dedupe_by_thread` 不再丢弃
2. 确认合并后的 snippet 包含所有消息的 subject + snippet
3. 启动 dev server，执行增量扫描，检查生成的卡片是否覆盖多个关注点
4. 回归：单封邮件的 thread 行为不受影响
